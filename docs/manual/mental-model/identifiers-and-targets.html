Glossary

Identifiers and Targets

TkVillage uses small string ids to connect declarations, live windows, reusable instances, and routing targets.

The Identifier Family

NameLooks LikeMeaning
window_kindmain, asset-browserAn app-chosen registered kind of window. It names a recipe, not a live window.
window_idwindow-1, window-2A TkVillage-generated id for one live Toplevel window.
instance_keymain:singleton, editor:abcA reuse key derived from the window kind and summon policy.
keyabc, C:/notes/today.mdAn app-supplied identity used when summoning a per-key window.
target_idwindow-1, autosaveA routing address for events and messages.

Window Ids

You do not pick a window_id. TkVillage generates it when it realizes a window. Current ids are generated from the window prefix, so they look like window-1, window-2, and so on.

summon_window() returns the live window record, whether it created a new window or reused an existing singleton/per-key window.

record = village.summon_window("main")
window_id = record["window_id"]

Window Targets

Every live window is also a target. For a window, the target id is the same string as its window id.

record = village.summon_window("main")
assert record["target_id"] == record["window_id"]

village.post_event(record["window_id"], {"type": "REFRESH"})

The helper functions let you avoid thinking about target ids when a more specific route is clearer.

  • post_event_to_window(window_id, event) posts directly to one live window.
  • post_event_to_instance(instance_key, event) resolves an instance key to a live window.
  • post_event_to_kind(window_kind, event) targets a singleton window kind.

Service Targets

For service targets, you choose the target_id when you register the service. Pick a stable, descriptive, unique string inside your app.

village.register_service_target("autosave", handle_autosave, {})
village.send_message("autosave", {"path": "notes.json"})

Typical service target ids look like autosave, filetalk-inbox, worker-status, app-coordinator, or debug-log. TkVillage does not impose a namespace beyond requiring the id to be the key under which the target is registered.

Instance Keys

Instance keys are how TkVillage knows when a summon should reuse a live window.

MultiplicityInstance Key ShapeReuse Rule
singleton<window-kind>:singletonOne live window for the whole kind.
per-key<window-kind>:<key>One live window for each app-supplied key.
free-instance<window-kind>:instance-1Each summon creates a new live window.
If you are unsure which id to use for routing to a window, start with the record returned by summon_window(). Its window_id is the window's routing address.
